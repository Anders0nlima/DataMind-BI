# ADR-001: DuckDB In-Memory Sandbox for Tabular Ingestion

**Status:** Accepted  
**Date:** 2026-06-28  
**Decision Makers:** Anderson (Tech Lead)  

---

## Context

DataMind BI must ingest user-uploaded spreadsheets (`.csv`, `.xlsx`) and allow an AI (Gemini 2.5 Flash) to answer analytical questions about the data. Three fundamental constraints shape this decision:

1. **Data Privacy:** Raw business/academic data must **never** leave the server. Sending 50,000 rows to an external LLM API is both a security risk and a cost disaster.
2. **Mathematical Accuracy:** LLMs hallucinate numbers. Any arithmetic must be performed by a deterministic engine, not by the language model.
3. **Performance:** Users expect sub-second responses for tabular operations on datasets up to 100MB.

## Decision

We will use **DuckDB** as an in-memory analytical SQL engine to:

1. **Ingest** uploaded files into an ephemeral RAM-based relational database (one DuckDB instance per user session).
2. **Extract schema** metadata (column names, data types, 3 sample rows) and send *only this schema* to Gemini.
3. **Execute SQL** queries generated by Gemini locally, returning exact numeric results to the AI for natural-language narration.

This implements the **Code Interpreter pattern**: `User Question → LLM writes SQL → Backend executes SQL → LLM narrates exact result`.

## Alternatives Considered

### Alternative A: Pandas DataFrame in Memory

| Criterion | DuckDB | Pandas |
|-----------|--------|--------|
| Query language | Standard SQL (Gemini writes SQL natively) | Python API (would require Gemini to write Pandas code — harder to sandbox) |
| 100MB CSV load time | ~0.2s | ~3–5s |
| Memory efficiency | Columnar, compressed | Row-major, uncompressed |
| SQL injection surface | Limited (read-only queries) | Arbitrary code execution risk |

**Rejected:** Pandas requires the LLM to generate Python code instead of SQL, which is harder to sanitize and has a larger attack surface.

### Alternative B: Send All Data to the LLM

**Rejected immediately.** Sending 50,000 rows to Gemini would:
- Cost ~$0.15–0.50 per query in input tokens.
- Hit context window limits.
- Allow the LLM to "do math" — violating our zero-hallucination principle.

### Alternative C: PostgreSQL / SQLite

**Rejected.** PostgreSQL adds deployment complexity (separate server process). SQLite lacks analytical query optimizations. DuckDB runs in-process, requires zero configuration, and is purpose-built for OLAP workloads.

## Consequences

### Positive
- **Zero hallucination on numbers:** All math is deterministic SQL execution.
- **Data sovereignty:** Raw data stays on the server RAM; only schema leaves.
- **Speed:** DuckDB ingests 100MB CSVs in <1 second.
- **Simplicity:** No external database server to deploy or manage.
- **LLM cost control:** Only schema (~500 tokens) is sent per query, not data rows.

### Negative
- **Session-scoped:** Data is lost when the server process restarts. This is acceptable for the current MVP (no persistent analytics).
- **Single-node limit:** DuckDB runs in-process. Horizontal scaling would require sharding sessions across workers (future concern).
- **SQL sanitization required:** We must validate that AI-generated SQL is read-only (`SELECT` only). See Commit 19 — SQL Sanitizer.

### Risks
- **Malicious SQL injection via LLM output:** Mitigated by a regex-based SQL sanitizer that blocks `DROP`, `DELETE`, `ALTER`, `INSERT`, `UPDATE`, and `CREATE` statements.

## Implementation Notes

- Each uploaded file creates a new DuckDB connection (`duckdb.connect()` with no file path = in-memory).
- Schema extraction returns: `{"columns": [{"name": "revenue", "type": "DECIMAL", "sample": [450.50, 320.00, 180.75]}]}`.
- The `openpyxl` library is required for `.xlsx` support via DuckDB's `read_xlsx()` or manual conversion.
